# Release AIMS-Core

Manage the aims-core release lifecycle: commit, PR, tag, deploy.

## When to use

When the user says "release aims", "tag aims", "deploy aims version", "aims release status", "cut a release", "prepare aims release", or invokes `/release-aims` with a subcommand.

## Subcommands

Parse the user's input to determine which subcommand to run:

| Trigger | Subcommand |
|---------|------------|
| `/release-aims status` or just `/release-aims` | STATUS |
| `/release-aims commit` | COMMIT |
| `/release-aims pr` | PR |
| `/release-aims tag` or `/release-aims tag 4.1.0` | TAG |
| `/release-aims deploy staging` or `deploy production` | DEPLOY |
| `/release-aims full` | FULL |

## Configuration

- **aims-core repo**: `/Users/kamir/GITHUB.active/my-ai-X` (git root), app at `aims-core/`
- **VERSION file**: `aims-core/VERSION` (contains semver like `4.0.0`)
- **State file**: `.deploy/gcp-state.json` in m3c-tools project root
- **Release strategy**: See `m3c-tools-maintenance/SPEC/release-strategy.md` if it exists
- **Image tag format**: `:latest`, `:v4.X.Y`, `:v4.X.Y-YYYYMMDD`

## How to execute

### STATUS subcommand

Gather and display release state:

```bash
# 1. Current branch and version
cd /Users/kamir/GITHUB.active/my-ai-X
BRANCH=$(git branch --show-current)
VERSION=$(cat aims-core/VERSION 2>/dev/null || echo "unknown")

# 2. Uncommitted changes count
CHANGES=$(git status -s | wc -l | tr -d ' ')

# 3. Latest tag
LATEST_TAG=$(git tag --sort=-v:refname | head -1)

# 4. Recent commits since main
COMMITS_AHEAD=$(git rev-list --count main..HEAD 2>/dev/null || echo "0")

# 5. Open PRs
gh pr list --repo GeoQB/my-ai-X --state open --limit 5
```

Read `.deploy/gcp-state.json` from m3c-tools to get deployment timestamps.

Display a summary table:
```
AIMS-Core Release Status
========================
Branch:          FR-user-roles
Version:         4.0.0
Latest tag:      v4.0.0
Uncommitted:     0 files
Ahead of main:   8 commits
Open PRs:        #27 (FR-user-roles → main)

Deployments:
  stage-legacy:  never deployed
  stage-new:     never deployed
  production:    never deployed
```

### COMMIT subcommand

Group uncommitted changes on the current feature branch into logical commits.

1. Run `git status -s` in the aims-core repo
2. If no changes, report "Nothing to commit" and stop
3. Analyze the changed files and group them by:
   - Bug fixes (files matching known BUG-XXXX patterns)
   - Feature additions (new files, new modules)
   - Chore/cleanup (config, Docker, deps)
   - Fix (corrections to existing behavior)
4. For each group, stage the relevant files and create a commit with a conventional commit message
5. Show the user the commits created

Follow the commit message conventions:
- `fix:` for bug fixes
- `feat:` for new features
- `chore:` for maintenance/deps/config
- `docs:` for documentation only

### PR subcommand

Create a PR from the current branch to main.

1. Check if a PR already exists: `gh pr list --head $(git branch --show-current) --base main`
2. If PR exists, show its URL and status
3. If no PR exists:
   - Run `git log --oneline main..HEAD` to get all commits
   - Auto-generate a PR body summarizing the commits
   - Create the PR:
     ```bash
     gh pr create --base main --head $(git branch --show-current) \
       --title "<branch-name>: <summary>" \
       --body "<auto-generated summary>"
     ```
4. Push the branch first if needed: `git push origin $(git branch --show-current)`

### TAG subcommand

Tag a release on main after PR merge.

1. If a version argument is provided (e.g., `/release-aims tag 4.1.0`), use it
2. Otherwise, read `aims-core/VERSION` for the version
3. Verify we're on main or can checkout main:
   ```bash
   cd /Users/kamir/GITHUB.active/my-ai-X
   git checkout main
   git pull origin main
   ```
4. Update the VERSION file if a new version was specified:
   - Edit `aims-core/VERSION` with the new version
   - Commit: `chore: bump version to X.Y.Z`
   - Push to main
5. Create and push the tag:
   ```bash
   VERSION=$(cat aims-core/VERSION | tr -d '[:space:]')
   git tag -a "v${VERSION}" -m "Release v${VERSION}"
   git push origin "v${VERSION}"
   ```
6. Report the tag URL

### DEPLOY subcommand

Build and deploy to the specified environment. This delegates to the `/deploy-gcp` skill.

1. Parse the target environment from the argument: `staging` maps to `stage-legacy`, `production` maps to `production`
2. Read the current version from `aims-core/VERSION`
3. Invoke the deploy-gcp skill workflow with the target environment
4. After successful deploy, update `.deploy/gcp-state.json`:
   - Set `current_version` to the version from VERSION file
   - Set the environment's `version_deployed` to the same version
   - Set `last_deployed` to the current ISO timestamp

### FULL subcommand

Orchestrate the complete release pipeline. Ask the user to confirm before each major step.

1. **Commit** — Run COMMIT subcommand
2. **Push** — Push the branch: `git push origin $(git branch --show-current)`
3. **PR** — Run PR subcommand
4. **Wait** — Tell the user: "PR created. Review and merge it, then say 'continue' to proceed with tagging."
   - Use AskUserQuestion to wait for confirmation
5. **Tag** — Run TAG subcommand (checks out main, pulls, tags)
6. **Deploy staging** — Run DEPLOY with `stage-legacy`
7. **Verify** — Check health endpoint, report status
8. **Deploy production** — Ask: "Staging looks good. Deploy to production?" If yes, run DEPLOY with `production`


## Image tagging

When building images, use these tags:
- `:latest` — always points to newest build
- `:v${VERSION}` — e.g., `:v4.0.0`
- `:v${VERSION}-$(date +%Y%m%d)` — dated snapshot, e.g., `:v4.0.0-20260313`

## Important notes

- Staging deploys use `cloudbuild-app.yaml` for fast builds (~3 min) when available
- Production deploys use manual `tools/v4/build_and_deploy.sh production`
- Always deploy to staging first, verify, then promote to production
- The VERSION file is the single source of truth for the release version
- Build scripts (`build_images_multi_platform_v4.sh`, `build_and_deploy.sh`) read from VERSION
- Production deployments ALWAYS require explicit user confirmation
- After tagging, do NOT delete or move the tag without user approval
- The `gcp-state.json` file tracks what version is deployed where
