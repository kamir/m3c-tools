Architectural intent — system insights as an integration space

This form is not just a checklist. The output is an ER1 insight document that closes the loop between system events and the personal memory layer:

  1. Capture-side: the trainer/operator runs the gate, observes outcomes, types feedback, and submits. The form emits one m3c-system-insight/v1 item per step plus one summary item for the gate as a whole.
  2. Storage-side: these items live in ER1 with the same shape as personal memos — the integration space — but tagged so they can be filtered.
  3. View-side: the thinker's main view (memory landing page, dashboard) filters out auto:generated + system:insight by default. Toggling "Show system insights" surfaces them when the thinker explicitly wants the bird's-eye cross-system view. The thinker is never overloaded with strangers' thoughts, but the system's voice is reachable inside the thinker's space.

Tag convention

Every insight item carries at least one dimension tag from each axis:

Origin axis: personal:thought system:insight system:decision auto:generated

Source axis (for system items): release-gate retro drift-check manifest-audit security-scan audit-replay

Subject axis: skill-manager aims-core m3c-tools kup-berlin …

Filter rule (for the thinker's main view)

visible_by_default = NOT (tags ∋ "auto:generated"
                          OR tags ∋ "system:insight")
override = explicit "Show system insights" toggle

Wiring it back

The form's Download JSON output matches the m3c-system-insight/v1 schema. Push it via:

./push-to-er1.sh release-gate-2026-05-07.json

The companion script (next to this form) takes the JSON, splits it into one ER1 item per step, applies the right tags, and posts to the existing /api/v2/memory/items endpoint. Each step is an item; the verdict + rationale survive as searchable text. The cross-step relationship survives via a shared session_id tag, so the thinker can pull the whole gate's worth of insights back as one bundle when they want it.
